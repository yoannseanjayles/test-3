## 2. Pipeline vidéo (système + encodage)

**Choix proposé : pipeline maison ffmpeg → HLS statique** (100 % open source, 0 € de licence).

```
Source (Internet Archive / open movie, MP4)
   │  1. Contrôle qualité + curation (back-office, H12)
   ▼
ffmpeg (batch, script versionné)
   │  H.264 High + AAC-LC · GOP 2 s · segments 6 s
   │  Ladder : 1080p ~5 Mbps · 720p ~2,8 Mbps · 480p ~1,2 Mbps
   │  + piste audio seule · + miniatures de scrubbing (sprites WebVTT)
   ▼
Artefacts HLS statiques
   │  master.m3u8 + renditions + segments .ts/.fmp4 + sous-titres .vtt
   ▼
Stockage objet + CDN (voir §3)
```

Justifications :
- **HLS seul** (pas de DASH) : compatibilité universelle 2026, aucun besoin DRM en v1 (D3/§A.1 de l'addendum validé) ;
- **3 qualités suffisent** : les sources domaine public dépassent rarement le 1080p ; chaque qualité ajoutée ≈ +50 % de stockage ;
- **Encodage batch hors ligne** (pas de service d'encodage à la volée) : le catalogue visionnable est curaté et de taille modérée — le temps réel n'apporte rien et les services managés facturent l'encodage (Mux : ~0,07 $/min) ;
- Script d'encodage **versionné dans le repo** : reproductible, ré-encodage global possible (changement de ladder, passage AV1 plus tard).

## 3. Hébergement / diffusion — comparatif et choix

| Solution | Coût réel | Encodage | Forces | Faiblesses |
|---|---|---|---|---|
| **Cloudflare R2 + HLS statique** ✅ | **10 GB gratuits puis ~0,015 $/GB/mois ; egress 0 $ illimité** | Le nôtre (ffmpeg) | Zéro coût de diffusion quel que soit le trafic, contrôle total, S3-compatible, se met derrière le CDN/Workers Cloudflare gratuits | Pipeline à opérer nous-mêmes |
| Mux (free tier) | 100 k min délivrées/mois VOD, assets stockés limités ; ensuite 0,07 $/min encodage + 0,025 $/min delivery | Inclus | « Developer-first », analytics par seconde, URLs signées natives | Le hors-quota coûte cher (l'encodage surtout) ; dépendance forte |
| Cloudflare Stream | 100 min stockage/10 k min delivery offerts avec plan Pro/Business, sinon 5 $/1 000 min stockées + 1 $/1 000 min | Inclus | Simplicité maximale (upload → HLS + player) | Pas de vrai free tier sans plan payant ; stockage à la minute coûteux pour des longs métrages |
| Bunny Stream | Essai 14 j puis ≥ 1 $/mois | Inclus | Le moins cher à gros volume | Pas gratuit |
| VPS self-host complet | Coût VPS + bande passante | ffmpeg | Contrôle total | La bande passante vidéo tue un VPS ; ops lourdes |

**Choix proposé : Cloudflare R2 (stockage) + CDN Cloudflare (diffusion) + Workers (URLs signées).**

Dimensionnement honnête : un long métrage 90 min en 3 qualités ≈ **6 GB**. Le free tier (10 GB) couvre ~1-2 films — c'est le **PoC**. Un catalogue curaté de 50 films ≈ 300 GB ≈ **4,50 $/mois** de stockage, **0 $ de diffusion** quel que soit l'audience. C'est le seul modèle du marché où le coût ne croît pas avec le succès (l'egress gratuit est la variable qui explose partout ailleurs). À titre de comparaison, 50 films sur Stream ≈ 22 $/mois de stockage + 1 $/1 000 min vues.

**Mux free tier reste l'alternative managée documentée** (bascule possible : le lecteur consomme un `master.m3u8` dans les deux cas). Les **bandes-annonces restent sur YouTube embed** (D5) — elles ne passent jamais par notre pipeline.

## 4. Lecteur vidéo

**Choix proposé : moteur `hls.js` + couche UI `Vidstack`** (tous deux open source).

| Critère | Justification du choix |
|---|---|
| Poids | hls.js ~80 KB — le plus léger pour du HLS pur (Shaka ~300 KB, surdimensionné sans DRM/DASH) |
| UX premium | Vidstack : composants modernes, thémables (design system Phase 4), React-ready |
| Accessibilité | Vidstack est accessible par construction (clavier, ARIA) — exigence WCAG 2.2 (D1) |
| Safari/iOS | HLS natif utilisé automatiquement (pas de MSE nécessaire) |
| Évolutivité | Migration DRM v2 = remplacement du moteur (Shaka) sous la même UI, pas de refonte |

**Contrôles v1** (standards validés D3) : lecture/pause · ±10 s · vitesse (0,5-2×) · sélecteur qualité (auto + manuel) · volume · sous-titres · PiP · plein écran · raccourcis clavier (`Espace`, `←/→`, `F`, `M`, `C`) · miniatures de scrubbing · `prefers-reduced-motion` respecté.

**Slot publicitaire (D6)** : le lecteur expose un point d'extension `adSlot` **abstrait** — pipeline : `config pub (remote) → si pub activée ET consentement TCF → charge le module pré-roll (IMA SDK, lazy-loadé) → sinon lecture directe`. Publicité désactivée = le module pub n'est **jamais chargé** (zéro octet, zéro requête). L'interrupteur est côté serveur (config runtime), effet immédiat sans redéploiement.

**Écran de fin** : recommandations post-visionnage annulables + compte à rebours épisode suivant (séries).

## 5. Sous-titres et langues

- **Format WebVTT** (standard HLS), pistes multiples déclarées dans le master playlist (`SUBTITLES`) ;
- Sources : fichiers `.srt` d'Internet Archive convertis en `.vtt` dans le pipeline ffmpeg ; open movies fournis avec sous-titres multilingues ;
- **Personnalisation** (exigence accessibilité) : taille, fond, contraste — préférences stockées au profil utilisateur (Paramètres) ;
- Langues v1 : FR + EN + VO d'origine quand disponible ; l'UI du lecteur suit la langue du site (FR, structure i18n préparée — H3/H6).

## 6. Reprise de lecture et historique de visionnage

### Modèle de données

```
playback_progress(user_id, title_id, episode_id?, position_s, duration_s,
                  updated_at, device_hint)         -- 1 ligne par utilisateur × contenu
watch_history(user_id, title_id, episode_id?, event, at)
                  -- events : started, progressed(25/50/75), completed
```

### Mécanique (standards D3)

- **Heartbeat** : `POST /api/progress` toutes les **15 s** de lecture effective (pas en pause), throttlé, + `navigator.sendBeacon` sur fermeture/onglet caché ;
- **Seuils** : < 5 % ou < 2 min → on repropose du début ; > 92 % → marqué **vu** (event `completed`), retiré de « Continuer la lecture » ;
- **Cross-device** : la reprise lit la position serveur au chargement du lecteur ; l'écriture la plus récente gagne (horodatage) ;
- **« Continuer la lecture »** : rail n°1 de l'accueil (ordre canonique D1), trié par `updated_at` desc ;
- **RGPD** : l'historique est une donnée personnelle — purgeable par l'utilisateur (page Historique, D4), exportable, TTL configurable ; les événements alimentent recos (Phase 6) et analytics (Phase 9) sous consentement.

## 7. Sécurité de diffusion

| Mesure | Implémentation |
|---|---|
| **URLs signées à expiration** | Worker Cloudflare devant R2 : token HMAC (user/session + titre + expiration ~6 h) vérifié sur le master playlist **et** les segments |
| Anti-hotlink | Vérification `Referer`/`Origin` au niveau Worker + règles CDN |
| Rate limiting | Par session sur l'émission de tokens (empêche l'aspiration du catalogue) |
| Clés & secrets | Signature côté serveur uniquement (principe D1 : rien dans le client) |
| DRM | **Non requis en v1** (contenu domaine public/CC) ; trajectoire v2 : moteur Shaka + Widevine/FairPlay sous la même UI (§4) |
| Conformité licences | Mention source + licence sur chaque fiche visionnable (D5 §3.3) ; le pipeline n'ingère que des contenus dont la licence est vérifiée en curation |

## 8. Architecture d'ensemble

```
                        ┌────────────── Lecteur (hls.js + Vidstack) ──────────────┐
                        │  adSlot (D6: OFF par défaut) · WebVTT · heartbeat 15 s  │
                        └──────┬──────────────────────────────┬───────────────────┘
                               │ master.m3u8 + segments        │ /api/progress · /api/token
                               ▼                               ▼
              CDN Cloudflare + Worker (URLs signées)      BFF / API interne
                               │                               │
                               ▼                               ▼
                        R2 (artefacts HLS)          PostgreSQL (progress, history)
                               ▲
                    Pipeline ffmpeg (batch, curaté)
                               ▲
              Internet Archive · Open movies (licences vérifiées)
```

## Hypothèses signalées

- **H12 (rappel, à lever ici)** : un **PoC du pipeline** (1 film Internet Archive + 1 open movie ré-encodés → R2 → lecteur) est prévu comme première tâche de la Phase 5/6 pour valider la qualité réelle avant industrialisation.
