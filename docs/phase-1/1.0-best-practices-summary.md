# Phase 1.0 — Best Practices Summary

> **Livrable** : Synthèse des meilleures pratiques (état de l'art 2025-2026)
> **Statut** : ⏳ En attente de validation HITL
> **Sources étudiées** : Netflix, Disney+, Prime Video, Apple TV, Plex, TMDB, JustWatch, Letterboxd, IMDb, Trakt + standards NN/g, Baymard, W3C/WCAG 2.2, web.dev, Google Search Central.

Cette synthèse couvre les 5 axes exigés par la phase 1.0 — **architecture, navigation, catalogue, recherche, recommandations** — complétés par les axes transverses obligatoires (SEO, accessibilité, performance). L'objectif est d'identifier les pratiques des leaders **sans les copier**, pour éclairer les décisions des étapes 1.1 à 1.4.

---

## 1. Architecture d'une plateforme de découverte

### Ce que font les leaders

| Plateforme | Enseignement clé |
|---|---|
| **Netflix** | Fédération GraphQL (70+ microservices), pages d'accueil **précalculées** servies par cache EVCache (>99 % hit rate). Leçon : précalculer les rails plutôt que les composer à la requête. |
| **Letterboxd** | Stack volontairement simple : monolithe Java + PostgreSQL + Redis (métadonnées) + Varnish (pages) + CDN Cloudflare. Preuve qu'un catalogue « taille moyenne » n'a pas besoin de microservices. |
| **JustWatch** | Valeur = **fraîcheur de la donnée** (refresh < 24 h) exposée via API GraphQL. La donnée différenciante prime sur la complexité technique. |

### Pratiques à retenir

1. **Rendu hybride ISR** (Incremental Static Regeneration) : fiches film/série générées statiquement et revalidées périodiquement (~24 h) ; SSR réservé aux pages dynamiques (recherche, listes utilisateur) ; CSR uniquement pour les interactions personnalisées. C'est le standard 2025-2026 des catalogues adossés à des API tierces.
2. **BFF (Backend For Frontend) obligatoire** : la clé API tierce ne vit **que côté serveur**. Le front n'appelle que nos routes. Erreur n°1 documentée du secteur : clé exposée dans le bundle JS.
3. **Cache à 3 étages** :
   - **CDN/edge** : `s-maxage` court + `stale-while-revalidate` + `stale-if-error` (survit aux pannes de l'API tierce) ;
   - **Redis cache-aside** avec TTL différenciés et « jitterés » (config/genres : 7 j · fiches : 24 h · trending : 1 h) ;
   - **Miroir local (PostgreSQL)** des entités consultées, alimenté incrémentalement (TMDB : Daily ID Exports + endpoint `/changes`).
4. **Protection anti-stampede** : request coalescing (une seule requête concurrente identique vers l'API tierce), TTL désynchronisés.
5. **Pipeline d'images** : proxy + transformation à la volée (AVIF → WebP → JPEG), srcset 3-5 breakpoints, jamais de hotlinking naïf des images du fournisseur.
6. **Modèle de données canonique** : `titles` (film/série) → `seasons` → `episodes` ; `people` + `credits` (jointure rôle/ordre) ; `genres` N-N ; IDs externes (`tmdb_id`, `imdb_id`) en colonnes indexées.
7. **Pagination par curseur** (keyset) pour l'infinite scroll — jamais d'OFFSET profond (dégradation linéaire + doublons).

### Erreurs fréquentes à éviter

- Clé API dans le code client ; proxy « passe-plat » sans rate limiting propre (quota épuisé par un seul client) ;
- 1 requête API tierce par requête utilisateur (aucun cache serveur) ; re-fetch N+1 par item de liste ;
- Cache HTTP `public` sur des réponses personnalisées (fuite de données entre utilisateurs) ;
- Rate limiting par IP seule (casse derrière NAT).

## 2. Navigation

### Constats (leaders 2024-2026)

- **Convergence vers une nav horizontale simple de 5-7 entrées** : Prime Video a abandonné sa sidebar (2024), Disney+ est passé à la barre horizontale (2025). Méga-menu uniquement si catalogue très profond (cas IMDb).
- **Mobile : bottom tab bar 4-5 items**, pas de hamburger pour le primaire (fonctions cachées = -30 à -50 % de découverte).
- **Ne pas transposer une UI TV au web** : au clavier/souris on attend de vrais liens `<a>`, des URLs propres et des pages de catégorie navigables — c'est l'avantage du web sur les apps TV et un anti-pattern fréquent des « clones de Netflix ».
- Recherche **toujours visible** (icône + raccourci clavier `/`), jamais enfouie.

### Recommandations

- Nav globale : Accueil · Films · Séries · Découvrir/Genres · Ma liste · Recherche.
- URLs propres et crawlables pour chaque section et chaque état de filtre pertinent.

## 3. Catalogue (browse)

### Constats

- **Hero** : 1 seul titre mis en avant, jamais d'auto-défilement sans pause (NN/g : ~1 % d'interaction avec les carrousels-bannières, 89 % sur la 1re slide).
- **Rails horizontaux** : standard du secteur, mais 8-12 max, ordre canonique (Reprise → Tendances/Nouveautés → personnalisés → thématiques), **libellés explicites** (« Parce que vous avez aimé Dune » > « Recommandé pour vous »).
- **Chaque rail doit pointer vers une vraie page grille filtrable** (« Voir tout ») — c'est là que les sites de découverte (Letterboxd, JustWatch, TMDB) surclassent les streamers.
- **Filtres facettés** : 5-7 facettes (genre, année/décennie, note, plateforme, type, durée), compteurs par valeur, chips de filtres actifs retirables, état dans l'URL. Baymard : 78 % des sites mobiles ont un filtrage médiocre → **opportunité de différenciation majeure**.
- **« Load More » plutôt qu'infinite scroll pur** ou pagination classique (Baymard) ; préserver la position de scroll au retour depuis une fiche ; conserver des URLs paginées crawlables.

### Erreurs fréquentes

- Surcharge de rails et paralysie du choix (« Netflix Syndrome » : ~17 min par choix, documenté académiquement) ;
- Mêmes titres répétés dans 5 rails → catalogue perçu comme pauvre ;
- Autoplay sonore (backlash Netflix : pétition à 120 000+ signatures) ;
- Carrousels auto-défilants inaccessibles (perte de focus clavier).

### Fiches détail (film/série)

Hiérarchie éprouvée : **hero visuel (backdrop) → titre + année + durée + genres + classification → note agrégée → CTA (bande-annonce · + Ma liste · Où regarder) → synopsis → casting avec photos → médias → avis → titres similaires**.
- Bande-annonce en 1 clic, dans une modale, jamais en autoplay sonore.
- Module **« Où regarder »** par pays et type d'offre (abonnement/location/achat) avec logos — modèle JustWatch, devenu l'élément central de la décision.
- Séries : sélecteur de saisons, progression d'épisodes, badges (« Nouvel épisode », « Finale »).

## 4. Recherche

### Constats

- **Recherche instantanée** : débounce 200-300 ms, annulation des requêtes en vol, cache des requêtes déjà effectuées, résultats perçus < 100 ms.
- **6-8 suggestions max** (Baymard) avec **vignette (affiche) + année + badge type** (film/série/personne), navigation clavier complète. Seuls 19 % des sites implémentent tous les détails correctement.
- **> 30 % des requêtes contiennent une faute** → tolérance aux fautes indispensable (« Vouliez-vous dire… »).
- **Jamais zéro résultat** (pattern Netflix) : en cas d'absence de correspondance, afficher tendances + titres proches + correction orthographique + réinitialisation des filtres.
- TMDB `/search/multi` fonctionne comme autocomplétion gratuite (frappe partielle, ~75 ms) mais **sans typo-tolérance** — à compenser (normalisation accents/casse, petit index client Fuse.js des titres populaires pour le « Vouliez-vous dire »).
- TMDB `/discover` = moteur à facettes gratuit (30+ filtres, opérateurs AND/OR) permettant toute la navigation filtrée sans base propre au départ.

### Stack recherche petit budget (règle de décision)

| Besoin | Solution |
|---|---|
| Catalogue complet, coût zéro | Proxy TMDB search + cache serveur des requêtes populaires |
| < ~5 000 titres (watchlist, catalogue curaté) | Index client (Fuse.js) — latence nulle |
| Typo-tolérance + facettes + classement custom sur données propres | Meilisearch ou Typesense self-hosted (0 € sur VPS) |

## 5. Recommandations

### Ce que font réellement les plateformes

- **TMDB** : préférer `/recommendations` (collaboratif précalculé sur les usages TMDB, nettement meilleur) à `/similar` (simple similarité genres/keywords) — `/similar` en secours si vide.
- **Letterboxd** : pas de moteur « Pour vous » — le **social + éditorial (listes, likes, activité)** remplace l'algorithme. Leçon : la curation a une forte valeur perçue à coût nul.
- **Trakt / JustWatch** : recommandations filtrées par les **services de streaming de l'utilisateur** — la disponibilité réelle est le premier filtre de pertinence.

### Boîte à outils sans budget ML (coût croissant)

1. **Trending/popularité** : endpoints `trending`/`popular`/`top_rated`, croisés avec `with_watch_providers` ;
2. **Similarité contenu maison** : score pondéré genres ∩ keywords ∩ cast/réalisateur ∩ année (réplicable en SQL sur miroir local) ;
3. **Co-occurrence item-item** (« les membres qui ont aimé A ont aussi aimé B ») : batch nocturne en SQL pur dès quelques centaines d'utilisateurs actifs ;
4. **Curation éditoriale** : listes thématiques manuelles — excellent pour le SEO et le cold start.

### Patterns de personnalisation

- **Onboarding « taste picker »** : 3 genres + 3 titres aimés + services de streaming à l'inscription (sinon la première session est du bruit) ;
- Rails **explicables** (« Parce que vous avez ajouté X ») — la transparence augmente la confiance (étude PMC) ;
- **Watchlist = signal central** : alimente les recs, les notifications de disponibilité et le filtre « Mes services » ;
- Toujours **exclure le déjà-vu / déjà-en-watchlist** des rails.

## 6. Axes transverses obligatoires

### 6.1 SEO (catalogue à grande échelle)

- **Rendu** : le CSR pur est disqualifié (indexation en 2 vagues, crawlers IA sans JavaScript). Hiérarchie : SSG ≥ ISR > SSR ≫ CSR. HTML initial complet : contenu, liens `<a>`, JSON-LD, meta/OG.
- **Données structurées** : JSON-LD `Movie`/`TVSeries` + `AggregateRating` (si la note est visible sur la page, sinon pénalité) + `BreadcrumbList` sur les fiches ; `ItemList` sur les listes.
- **URLs** : `/film/slug-annee` ou slug + ID stable (désambiguïse les remakes, survit aux renommages TMDB) ; canonical auto-référentiel ; pages paginées auto-canoniques (jamais vers la page 1).
- **Indexation** : indexer fiches + pages genre à demande réelle ; noindex recherche interne, tris, combinaisons de filtres ; jamais `noindex` + `Disallow` ensemble ; sitemaps segmentés par type (< 50 k URLs) avec `lastmod` fiables.
- **Anti-thin-content** (spam updates Google 2025) : chaque page doit agréger une **valeur unique** (dispo, notes, casting, fraîcheur) — c'est le modèle gagnant de JustWatch (SEO programmatique sur « où regarder X » avec donnée propriétaire fraîche).
- **Social** : `og:image` dynamique 1200×630 **composée** par titre (l'affiche 2:3 recadrée ne suffit pas) + Twitter Card.

### 6.2 Performance (Core Web Vitals)

- **LCP** (~62 % d'échec mobile, l'image en est la cause dans 76-85 % des cas) : affiche/hero en HTML avec `fetchpriority="high"` + preload, **jamais lazy-loadée** ; AVIF → WebP → JPEG ; tailles TMDB natives (`w185/w342/w500`) alignées sur les breakpoints.
- **CLS** : `aspect-ratio` 2:3 réservé sur toutes les affiches, placeholders flous (LQIP/blurhash), hauteur fixe des rails, transitions en `transform` uniquement.
- **INP** (≤ 200 ms) : `useTransition` pour filtres/recherche, code-splitting du lecteur de bande-annonce et des modales, virtualisation des longues listes, `content-visibility: auto` sur les rails hors écran.

### 6.3 Accessibilité (WCAG 2.2)

- **Cibles ≥ 24×24 px** (critère 2.5.8 nouveau) : flèches de rails, boutons watchlist sur cards ;
- **Focus visible et non masqué** (2.4.11 — attention aux headers sticky) ; les actions révélées au hover doivent aussi apparaître au focus ;
- **Carrousels** : vrais `<button>` précédent/suivant, pause obligatoire sur tout autoplay, `role="region"` + `aria-label` par rail ; un conteneur scrollable avec cards tabbables est souvent plus robuste qu'un carrousel ARIA complet ;
- **Dark mode ≠ dispense** : contraste 4,5:1 (texte) / 3:1 (UI) vérifié sur le thème sombre ; fond `#121212` (jamais `#000` pur), texte off-white ;
- **`prefers-reduced-motion`** respecté partout (autoplay, zooms hover, transitions) ;
- **Alt text** : `alt=""` sur l'affiche redondante en card (titre visible à côté), alt descriptif sur la fiche ; un seul lien englobant par card.

### 6.4 Design premium (tendances 2025-2026)

- **Dark-mode-first** (standard du secteur : les visuels sont la matière première du design) avec accent coloré unique ;
- **Skeleton screens** partout où il y a du fetch, shimmer subtil ; micro-interactions 150-250 ms ;
- **Hover preview** : agrandissement de carte différé (~500 ms), vidéo uniquement muette et opt-out — leçon du fiasco autoplay Netflix ;
- Tendance visuelle : matériaux translucides (« Liquid Glass » Apple tvOS 26), tuiles enrichies de badges contextuels (Netflix 2025).

## 7. Top 10 des principes directeurs pour la suite du projet

1. ISR + BFF + cache 3 étages : aucune clé API côté client, aucune requête API tierce non cachée.
2. Miroir local incrémental des données plutôt que proxy pur (curseurs, facettes, résilience, SEO).
3. Nav simple 5-7 entrées, bottom tab bar mobile, recherche proéminente.
4. Accueil : hero unique + ≤ 12 rails explicables, chacun avec un « Voir tout » vers une grille filtrable.
5. Filtres facettés d'excellence (compteurs, chips, URL) — le point faible du marché, notre différenciation.
6. Fiche titre : hiérarchie métadonnées → CTA → casting → similaires, module « Où regarder » neutre.
7. Recherche instantanée < 100 ms perçues, typo-tolérance, jamais zéro résultat.
8. Recommandations sans ML : trending + `/recommendations` TMDB + co-occurrence SQL + curation éditoriale, toujours explicables.
9. SEO programmatique légitime : JSON-LD, URLs stables, sitemaps segmentés, valeur unique par page.
10. Accessibilité WCAG 2.2 et Core Web Vitals intégrées dès la conception, pas en fin de projet.

## Hypothèses signalées

- **H1** : TMDB sera vraisemblablement l'API principale (gratuite, riche, images, endpoints discover/trending/recommendations) — le choix formel et la comparaison complète relèvent de l'étape **1.2**.
- **H2** : Les volumes visés (catalogue TMDB ≈ 1 M+ titres, trafic initial modeste) permettent une architecture simple (modèle Letterboxd) plutôt que microservices (modèle Netflix).
- **H3** : Le marché principal est francophone (interface FR), avec une structure préparée pour l'i18n (modèle sous-répertoires + hreflang).
