# Rapport de recherche — Architecture des plateformes de découverte de films/séries (état de l'art 2025-2026)

Périmètre : couche catalogue/découverte uniquement (type TMDB, JustWatch, Letterboxd, Trakt, IMDb, couche "browse" de Netflix), hors diffusion vidéo.

---

## 1. Patterns d'architecture front-end / back-end chez les leaders

**Constats clés**

- **Rendu hybride SSR/SSG/ISR** : le standard 2025-2026 pour les catalogues est Next.js (ou équivalent) avec ISR (Incremental Static Regeneration) pour les pages de détail film/série — le contenu change rarement, donc on génère statiquement et on régénère à intervalle configurable ; SSR est réservé aux pages dynamiques/personnalisées (recherche, listes utilisateur). C'est le pattern dominant des implémentations TMDB modernes ([dev.to — Building a Movie Platform with Next.js and TMDB](https://dev.to/mr_elvis/building-a-movie-platform-with-nextjs-and-tmdb-api-43gh), [raftlabs — Next.js Best Practices 2025](https://www.raftlabs.com/blog/building-with-next-js-best-practices-and-benefits-for-performance-first-teams), exemple open-source : [oktay/movies, client TMDB Next.js 14](https://github.com/oktay/movies)).
- **BFF (Backend For Frontend)** : Netflix est passé de BFF dupliqués par plateforme à une **fédération GraphQL** : un supergraph unique composé de sous-graphes (framework DGS, Spring Boot/Java), chaque domaine (catalogue, artwork, personnalisation) étant un microservice possédant sa tranche de schéma ; ~70+ microservices orchestrés, déploiements indépendants ([Netflix TechBlog — GraphQL Federation Part 1](https://medium.com/netflix-techblog/how-netflix-scales-its-api-with-graphql-federation-part-1-ae3557c187e2), [Apollo — Why Netflix chose Federated GraphQL](https://www.apollographql.com/blog/redefining-api-strategy-why-netflix-platform-engineering-chose-federated-graphql), [InfoQ](https://www.infoq.com/presentations/netflix-api-graphql-federation/)). Pour une petite plateforme, le BFF simple (routes API Next.js / petit gateway) suffit — la fédération est un pattern de très grande échelle.
- **Exemple concret de stack "taille moyenne" — Letterboxd** : backend Java (Tomcat + nginx), PostgreSQL + réplicas, **Redis pour le cache de métadonnées, Varnish pour le cache de pages**, CDN Cloudflare devant tout, hébergement AWS ([Quora — Letterboxd tech stack](https://www.quora.com/What-is-the-technology-stack-behind-Letterboxd), [StackShare](https://stackshare.io/letterboxd/letterboxd)). JustWatch expose une **API GraphQL publique** et rafraîchit ses données de disponibilité au moins toutes les 24 h (ingestion des feeds partenaires ~toutes les 6 h) ([JustWatch API](https://www.justwatch.com/us/JustWatch-Streaming-API), [docs d'intégration](https://apis.justwatch.com/docs/streaming_service/)).
- **Pipeline d'images** : le pattern 2025 est le redimensionnement à la volée par CDN (Cloudflare Image Resizing, next/image, imgproxy) : une seule image d'origine, transformations par paramètres d'URL (`width`, `format=auto`), négociation AVIF → WebP → original par requête, srcset sur ~4 breakpoints (400/800/1200/1600w), lazy loading hors viewport et preload de l'image LCP ([BlazingCDN — Cloudflare Image Best Practices](https://blog.blazingcdn.com/en-us/cloudflare-image-cdn-best-practices-webp-avif), [FrontendTools — Image Optimization 2025](https://www.frontendtools.tech/blog/modern-image-optimization-techniques-2025), [Transloadit — pipeline CDN](https://transloadit.com/devtips/building-an-image-optimization-pipeline-with-a-cdn/)). Gain typique : 50-80 % de payload.

## 2. Modélisation des données catalogue et synchronisation d'API tierces

**Modèle de données**

- Hiérarchie canonique : `Series 1—N Seasons 1—N Episodes` ; `Movie` entité séparée (ou table `Title` polymorphe) ; `Person` unique réutilisée via tables de jointure `Credits` (person_id + title_id + rôle/character + ordre) ; `Genre` en N—N. Viser la 3NF pour le cœur, mais dénormaliser les agrégats de lecture (note moyenne, nombre d'épisodes) ([Laracasts — DB design for a TV series app](https://laracasts.com/discuss/channels/laravel/database-design-for-a-tv-series-app-1?page=0), [Kaan Celiker — Designing the DB for a film/series review app](https://kaanceliker.medium.com/part-1-designing-the-database-with-mssql-for-a-film-series-review-application-06c035e78602)).
- Stocker les **IDs externes** (tmdb_id, imdb_id, trakt_id) comme colonnes indexées : c'est ce que fait Trakt, qui pivote entre IMDb/TMDB/JustWatch via des mappings d'IDs ([Trakt API](https://trakt.docs.apiary.io/)).

**Synchronisation TMDB (le cas type d'API gratuite)**

- **Limites** : ~40-50 req/s et 20 connexions simultanées par IP côté API ; `image.tmdb.org` n'est **pas** limité en débit, seulement en connexions simultanées (20) ([TMDB Talk — rate limits & caching](https://www.themoviedb.org/talk/52f3f24ac3a36801cd272f45), [TMDB Talk — Rate Limit and Caching](https://www.themoviedb.org/talk/684fccb6bac1f078e05c83a8)).
- **Cache autorisé et encouragé** : les CGU autorisent le cache local (y compris héberger les posters derrière votre propre proxy d'images) tant que les demandes de retrait sont honorées ; limite contractuelle ≈ 6 mois de rétention ([TMDB Talk — Cache API data](https://www.themoviedb.org/talk/5bd3a0fd0e0a2622da00b695)). Honorer les en-têtes `Cache-Control`/`ETag` renvoyés par TMDB (revalidation conditionnelle gratuite) ([client Swift TMDb](https://github.com/adamayoung/TMDb)).
- **Sync incrémentale** : TMDB publie des **Daily ID Exports** (fichiers JSON quotidiens, ~7-8 h UTC, sur files.tmdb.org) listant tous les IDs valides — évite de balayer l'espace d'IDs et de gaspiller le quota sur des 404 ; stratégie recommandée : diff des fichiers J/J-1 + endpoint `/changes` pour ne re-fetcher que les fiches modifiées ([TMDB — Daily ID Exports](https://developer.themoviedb.org/docs/daily-id-exports), [blog TMDB](https://blog.themoviedb.org/daily-id-file-exports-5e0507323932)).
- **TTL différenciés** (pratique constatée dans les clients/wrappers TMDB) : configuration/genres = jours-semaines ; fiches film/série = 24 h-7 j ; listes trending/popular = 30 min-quelques heures ; recherche = courte durée en mémoire ([tmdb.lorenzopant.dev — rate limiting/caching](https://tmdb.lorenzopant.dev/docs/getting-started/options/rate-limiting)).

## 3. Scalabilité et performance : couches de cache et pagination

**Cache multicouche (le pattern universel)**

1. **CDN/edge** avec `stale-while-revalidate` : en-tête type `CDN-Cache-Control: public, s-maxage=60, stale-while-revalidate=600, stale-if-error=86400` — frais 60 s, servi périmé jusqu'à 10 min pendant la revalidation en arrière-plan, survit aux pannes d'origine 24 h. Séparer `CDN-Cache-Control` (edge, agressif) de `Cache-Control` (navigateur, strict). Ne **jamais** mettre `public` sur des réponses personnalisées ([Jono Alderson — HTTP caching guide](https://www.jonoalderson.com/performance/http-caching/), [Fastly — stale docs](https://www.fastly.com/documentation/guides/concepts/cache/stale/), [Cloudflare — Revalidation](https://developers.cloudflare.com/cache/concepts/revalidation/), [Vercel CDN Cache](https://vercel.com/docs/caching/cdn-cache)).
2. **Redis en cache-aside** pour les réponses d'API tierces et agrégats — pattern de référence pour les workloads read-heavy type catalogue ([Redis — cache-aside](https://redis.io/docs/latest/develop/use-cases/cache-aside/)). Protections indispensables : **TTL avec jitter** (±10-20 %) pour désynchroniser les expirations, **request coalescing / single-flight** (une seule requête origin par clé, les autres attendent le résultat) et verrou distribué ou recomputation probabiliste anticipée contre le **cache stampede** ([Redis Patterns — Cache Stampede Prevention](https://redis.antirez.com/fundamental/cache-stampede-prevention.html), [OneUptime — Redis cache stampede](https://oneuptime.com/blog/post/2026-01-21-redis-cache-stampede/view)).
3. Référence extrême : Netflix **EVCache** (memcached distribué) sert les pages d'accueil personnalisées **précalculées** (nightly), >99 % de hit rate, latence 1-5 ms, >1 M RPS en réplication inter-régions ; leçon transposable : précalculer les rayons/listes plutôt que les composer à la requête, et accepter la cohérence éventuelle ([Netflix TechBlog — Caching for a Global Netflix](http://techblog.netflix.com/2016/03/caching-for-global-netflix.html), [EVCache wiki](https://github.com/netflix/evcache/wiki)).

**Pagination / infinite scroll**

- **Cursor-based (keyset) pour l'infinite scroll** : performance constante quelle que soit la profondeur (jusqu'à 17× plus rapide qu'OFFSET en profondeur) et immunité au "shifting data problem" (doublons/éléments manqués quand le jeu de données bouge). OFFSET reste acceptable pour de petits jeux statiques ou l'accès direct à la page N ([Milan Jovanović — Understanding Cursor Pagination](https://milanjovanovic.tech/blog/understanding-cursor-pagination-and-why-its-so-fast-deep-dive), [Gusto — API Pagination guide](https://embedded.gusto.com/blog/api-pagination/), [Engineering at Scale — Limit-Offset vs Cursor](https://engineeringatscale.substack.com/p/api-pagination-limit-offset-vs-cursor)). Curseurs composites (`created_at + id`) opaques (base64) et index couvrants requis. Nota : TMDB lui-même ne fournit que des pages numérotées — d'où l'intérêt d'un miroir local pour offrir de vrais curseurs.
- **Recherche instantanée** : sous ~50 M de documents, Typesense ou Meilisearch (typo-tolérance, facettes, search-as-you-type natifs) surclassent opérationnellement Elasticsearch, réservé aux besoins d'agrégations lourdes ([Typesense — comparatif](https://typesense.org/typesense-vs-algolia-vs-elasticsearch-vs-meilisearch/), [Meilisearch — Elasticsearch vs Typesense](https://www.meilisearch.com/blog/elasticsearch-vs-typesense)). Côté client : debounce de l'input de recherche.

## 4. Erreurs d'architecture courantes des sites catalogue adossés à des API tierces

1. **Clé API exposée côté client** : toute clé dans le bundle JS est lisible en DevTools en quelques secondes ; la parade n'est pas d'« obfusquer » mais de déplacer la clé dans un BFF/proxy serveur (variables d'environnement) ([GitGuardian — Stop Leaking API Keys: the BFF pattern](https://blog.gitguardian.com/stop-leaking-api-keys-the-backend-for-frontend-bff-pattern-explained/), [dev.to — 10 Rookie Mistakes That Leak Your API Keys](https://dev.to/sky98/10-rookie-mistakes-that-leak-your-api-keys-and-how-to-secure-them-13e8)).
2. **Proxy « passe-plat » sans quota** : un BFF sans rate limiting propre permet à un seul client d'épuiser le quota du fournisseur pour tout le monde (voire de faire bannir le compte) ([dev.to — mistakes integrating the OpenAI API, transposable](https://dev.to/backmesh/openai-api-key-safety-and-mistakes-in-client-side-code-3k2j), [issue CLIProxyAPI](https://github.com/router-for-me/CLIProxyAPI/issues/3467)). Un proxy non authentifié qui forwarde tout est « une façon compliquée d'exposer sa clé ».
3. **Marteler l'API tierce** : 1 requête TMDB par requête utilisateur (pas de cache serveur), re-fetch N+1 par item de liste au lieu d'`append_to_response`/batching, scan complet de l'espace d'IDs au lieu des Daily Exports + `/changes` ([TMDB — Daily ID Exports](https://developer.themoviedb.org/docs/daily-id-exports), [dev.to — Working with API Rate Limits](https://dev.to/altug_gokoglu/working-with-api-rate-limits-lessons-from-developer-mistakes-11en)).
4. **Pas de gestion du stampede** : TTL uniformes qui expirent en même temps → rafale synchronisée vers l'API tierce ou la DB ([Scalar Dynamic — The Cache Stampede Problem](https://scalardynamic.com/resources/articles/22-the-cache-stampede-problem)).
5. **Cache HTTP mal configuré** : réponses personnalisées mises en cache `public` sans `Vary` (fuite de données entre utilisateurs) ; ou à l'inverse aucun `s-maxage`/SWR, donc CDN inutile ([Jono Alderson — HTTP caching](https://www.jonoalderson.com/performance/http-caching/)).
6. **Hotlinking naïf des images tierces** sans proxy/redimensionnement : dépendance au CDN du fournisseur, pas de conversion AVIF/WebP, tailles inadaptées, et plafond de connexions simultanées (20 chez TMDB) ([TMDB Talk — image usage](https://www.themoviedb.org/talk/52f3f24ac3a36801cd272f45)).
7. **OFFSET profond pour l'infinite scroll** : dégradation linéaire + doublons ([dev.to — Page Numbers Lie](https://dev.to/mandy8055/page-numbers-lie-offset-vs-cursor-pagination-39f4)).
8. Rate limiting **par IP seule** côté proxy : casse derrière les NAT/CGNAT et se contourne facilement ; préférer par utilisateur/clé de session ([Zuplo — Don't rate limit by IP](https://zuplo.com/blog/dont-rate-limit-by-ip)).

## Recommandations actionnables (plateforme moderne adossée à une API gratuite type TMDB)

1. **Next.js (ou équivalent) avec ISR** : fiches film/série statiques régénérées (revalidate 24 h), listes trending en SSR + cache edge court, recherche côté client via le BFF.
2. **BFF obligatoire** : la clé TMDB ne vit que côté serveur ; le front n'appelle que vos routes. Ajouter authentification/session + rate limiting **par utilisateur** sur le BFF.
3. **Cache à 3 étages** : CDN (`s-maxage` 60 s + `stale-while-revalidate` 600 s + `stale-if-error` 86400 s sur les endpoints publics) → Redis cache-aside (TTL jittérés : config/genres 7 j, fiches 24 h, trending 1 h) → miroir PostgreSQL pour les entités consultées.
4. **Miroir local incrémental plutôt que proxy pur** : ingestion via Daily ID Exports + endpoint `/changes` de TMDB, en respectant la limite contractuelle de ~6 mois et les take-downs. Cela débloque curseurs, facettes, et indépendance vis-à-vis des pannes du fournisseur.
5. **Single-flight/request coalescing** sur chaque clé de cache pour ne jamais envoyer plus d'une requête concurrente identique vers TMDB.
6. **Modèle de données** : `titles` (film/série), `seasons`, `episodes`, `people`, `credits` (jointure avec rôle/ordre), `genres` N—N, colonnes `tmdb_id`/`imdb_id` indexées ; dénormaliser les compteurs d'affichage.
7. **Images** : proxy + transformation à la volée (next/image, Cloudflare Images ou imgproxy) devant `image.tmdb.org`, `format=auto` (AVIF→WebP), srcset 400/800/1200/1600w, lazy loading + preload du poster LCP, cache CDN long (immutable par URL versionnée).
8. **Pagination par curseur** (created_at + id, curseur opaque) pour l'infinite scroll sur votre miroir ; jamais d'OFFSET profond.
9. **Recherche** : Meilisearch ou Typesense alimenté par le miroir (typo-tolérance, facettes genre/année/note) plutôt que l'endpoint search de TMDB en direct ; debounce 200-300 ms côté client.
10. **Attribution et conformité** : afficher le logo/mention TMDB requis par les CGU ; prévoir un job de purge/refresh pour les entités supprimées (présentes dans les diffs des exports quotidiens).
