# Phase 7.0 — Best Practices Summary : Administration

> **Étape** : 7.0 (recherche) · **Statut** : ✅ Validé HITL (D40)
> **Périmètre recentré (D36)** : le back-office complet est en production (6.1 Lot 4) — la Phase 7 le fait passer de « fonctionnel » à « opérable au quotidien » : paramétrage fin, mesure, revenus, notifications, anti-abus.
> **S'appuie sur** : D6/D11 (pilotage), 6.2/D39 (palier publicitaire P-1), H88/H94/H95/H96 (dettes tracées)

## 1. Config runtime typée (lève H96)

La table `app_config` (booléens) devient un **registre typé** : `app_settings (key, type: boolean|number|string, value jsonb, updated_by, updated_at)` avec un module `lib/settings` validé par Zod par clé (défaut, bornes min/max, description). Premières clés numériques : `ugc.quota.videos_per_user` (défaut 5), `ugc.upload.max_bytes`, `ads.preroll.session_cap`, `moderation.sla_hours`. Écran back-office « Paramètres » généré depuis le registre (jamais de clé libre — uniquement les clés déclarées). Migration : les 7 flags booléens existants sont repris tels quels, API `flags.ts` inchangée.

## 2. Anti-abus — rate limiting en base (lève H88)

Fenêtre glissante sans Redis : table `rate_limits (bucket, subject, count, window_start)` + `lib/rate-limit.ts` (`limit(bucket, subject, max, windowSeconds)`), appliqué aux mutations sensibles : login/register (par IP + par e-mail), contact (IP), création d'upload (utilisateur), heartbeat (session). Nettoyage paresseux des fenêtres expirées à l'écriture. Réponses : message générique + `Retry-After` ; les admins ne sont jamais limités sur la modération.

## 3. Événements produit — socle unique (prépare la Phase 9 et P-1)

Une seule table `events (id, name, session_anon, user_id?, props jsonb, created_at)` alimentée côté serveur (actions/handlers — pas de script tiers, pas de consentement requis pour la mesure d'audience *exempte* si anonymisée ; sinon activée sous CMP). Nommage : `domaine.action` (`video.play`, `video.complete`, `search.zero`, `contact.submit`, `ad.impression`…). Les `ad_events` du 6.2 sont **ce même socle** avec le préfixe `ad.` — un seul pipeline, deux lectures (Revenus en Phase 7, Growth en Phase 9). Rétention : 13 mois, agrégats journaliers matérialisés pour les tableaux de bord.

## 4. Tableau Revenus (palier P-1 du 6.2, lève H95)

Écran back-office « Revenus » : impressions/complétion/skip par emplacement et par jour (à partir des `ad.*`), eCPM saisi par campagne (régie directe v1), projection mensuelle. Tant que la pub est OFF : écran présent avec état vide explicite (« la publicité n'est pas activée ») — cohérent avec la méthode des états honnêtes. `lib/ads/decision.ts` (house ads) est implémenté à ce palier, branché sur les AdSlots existants.

## 5. Notifications créateurs (lève H94)

- **In-app d'abord** : table `notifications (user_id, type, payload, read_at)` + cloche dans le header (badge non-lus) et liste dans le Studio — alimentée par la modération (publiée/refusée + motif) et la fin d'encodage ;
- **E-mail ensuite** (Resend free tier, H83) : mêmes événements, template minimal, opt-out dans les paramètres ; l'e-mail n'est jamais le seul canal (l'in-app fait foi) ;
- Extension naturelle : notifications takedown pour les admins (nouveau message ayant droit → cloche admin).

## 6. Opérations & runbook

`docs/OPERATIONS.md` : procédures nommées — takedown (SLA 72 h, priorité D11), panne d'encodage (relance back-office, quota GitHub Actions), rotation des secrets, restauration Neon (branche de sauvegarde), seuils d'alerte free tier (H81 : 80 % de 0,5 Go base / 10 Go R2 / 2 000 min Actions, vérifiés par un job hebdomadaire qui notifie les admins). Suppression de compte RGPD : action back-office (cascade `list_entries`, anonymisation des `videos` publiées).

## Hypothèses signalées

- **H101** : la mesure d'audience anonyme servie sans cookie (comptage serveur par session éphémère) est exempte de consentement (guidelines CNIL « mesure d'audience exemptée ») — à confirmer en revue juridique H46 ;
- **H102** : les agrégats Revenus v1 sont calculés à la lecture (volumes faibles) ; matérialisation planifiée quand `events` dépassera ~1 M de lignes ;
- **H103** : la cloche de notifications interroge le serveur au chargement (pas de WebSocket/SSE en v1 — plan gratuit Vercel) ; temps réel réévalué en Phase 9.
