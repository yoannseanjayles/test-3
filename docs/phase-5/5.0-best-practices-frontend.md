# Phase 5.0 — Best Practices Summary : Front-end

> **Livrable** : Synthèse des meilleures pratiques de développement front-end (développement, performance, SEO, accessibilité) et **proposition de stack**
> **Statut** : ⏳ En attente de validation HITL
> **Applique** : D1 (ISR+BFF, budgets CWV), D20 (lecteur), D24-D25 (tokens) — cette synthèse traduit les décisions validées en choix d'implémentation.

---

## 1. Framework & rendu — proposition de stack

Les exigences validées imposent pratiquement la réponse : **ISR** par page (D1), SSR pour les pages dynamiques, routes serveur pour le BFF (clé TMDB jamais côté client), i18n préparée (H3), images optimisées.

| Choix | Proposition | Justification |
|---|---|---|
| Framework | **Next.js 15 (App Router) + TypeScript strict** | Seul framework couvrant nativement ISR (`revalidate` par page + `revalidateTag` pour les fiches chaudes D15), React Server Components (HTML complet SEO), Route Handlers (BFF co-localisé), `next/image` (pipeline AVIF/WebP D1). Alternatives évaluées : Astro (excellent SSG mais îlots moins adaptés à l'app connectée Ma liste/Studio), SvelteKit (ISR moins mûre), Remix (pas d'ISR native). |
| Styles | **Tailwind CSS v4** consommant `design/tokens.css` | v4 est CSS-variables-first : nos tokens D25 deviennent le thème sans duplication ; utilitaires = pas de cascade fragile ; purge automatique (budget JS/CSS). |
| Composants | **Bibliothèque maison** sur primitives accessibles ([Radix UI](https://www.radix-ui.com) pour modal/menu/tabs) | Le design system D25 est spécifique (états, Advertising DS) ; Radix apporte l'ARIA/focus éprouvé sans style imposé. |
| Lecteur | **Vidstack (React) + hls.js** (D7) | Décision validée ; thème via tokens. |
| État client | **TanStack Query** (serveur-état : watchlist, progression) + Zustand (UI locale : lecteur, overlay recherche) | Cache/mutations optimistes (D19 favoris « optimistic UI »), invalidation ciblée ; pas de Redux (surdimensionné). |
| Validation | **Zod** partagé (schémas API BFF ↔ formulaires) | Une seule source de vérité des types. |

## 2. Architecture du code

```
src/
  app/                    # App Router : (marketing)/ (catalog)/ (user)/ (studio)/ (admin)/ + api/
  components/ui/          # design system (Button, Card, Rail, AdSlot…)
  components/features/    # composés métier (WatchProviders, EpisodeList, Player…)
  lib/tmdb/               # client API tierce (server-only), mappers → modèles canoniques
  lib/db/                 # accès miroir/PostgreSQL (server-only)
  lib/ads/                # pipeline pub (chargé dynamiquement si pub ON — D6/D14)
  styles/                 # tokens.css importé + tailwind
  i18n/                   # dictionnaires FR (structure préparée, H3)
```
- **`server-only`** importé dans lib/tmdb et lib/db : toute fuite vers le client casse le build (garde-fou D1 n°1) ;
- Modèles **canoniques** (`Title`, `Person`, `Episode`) : les composants ne voient jamais la forme TMDB brute (protège du R2 — changement de fournisseur) ;
- Feature flags lus côté serveur (`ads.enabled`, `ugc.upload.enabled`) et injectés — jamais de secret ni de logique de flag dans le bundle.

## 3. Performance (application des budgets D1/D17)

- **RSC par défaut, client uniquement aux îlots interactifs** (lecteur, recherche, facettes, favoris) — le JS embarqué est le premier ennemi de l'INP ;
- `next/image` + tailles TMDB alignées (`w342`→`sizes`), hero `priority` (LCP), blurhash en `placeholder` (A6) ;
- `next/dynamic` pour : lecteur (préchargé à l'intention — H31), TrailerModal, Lightbox, module pub (D6 : zéro octet si OFF), CMP ;
- Fonts self-hosted WOFF2 via `next/font` (zéro layout shift, `size-adjust` du fallback) ;
- Budgets CI : **LCP < 2,5 s · INP < 200 ms · CLS < 0,1 · JS initial < 170 KB gzip** par gabarit — mesurés par Lighthouse CI sur chaque PR, échec = merge bloqué.

## 4. SEO technique (application D1 §6.1)

Metadata API par gabarit (title/description/OG dynamiques D15) · JSON-LD injecté côté serveur (Movie/TVSeries/Person/ItemList/VideoObject) · sitemaps segmentés générés depuis le miroir (route handlers) · redirections 301 de slug (D15) · `generateStaticParams` pour le top catalogue, ISR à la demande pour la longue traîne.

## 5. Accessibilité (application D1 §6.3 — vérifiée en continu)

Radix pour les patterns à piège (modales, menus) · tests **axe-core automatisés par gabarit** (CI) + navigation clavier scriptée (Playwright) sur les 5 parcours critiques (V1-V4, U5) · `eslint-plugin-jsx-a11y` strict · revue manuelle NVDA/VoiceOver par lot de pages (checklist par composant D25 §5).

## 6. Qualité, tests, CI

| Niveau | Outil | Cible |
|---|---|---|
| Statique | TypeScript strict + ESLint + Prettier | tout |
| Unitaires | Vitest + Testing Library | lib/ (mappers, recos, capping pub D17 §3) et composants ui/ |
| E2E | Playwright | 6 parcours : découverte→fiche→watch (avec et sans pub), recherche, auth+watchlist, reprise, upload UGC, admin modération |
| A11y | axe-core (CI) | chaque gabarit |
| Perf | Lighthouse CI | budgets §3 |

CI GitHub Actions : lint → types → unit → build → e2e → lighthouse. `main` protégée, preview par PR.

## 7. Analytics & consentement (application D6/Phase 9)

Wrapper `track()` maison (événements typés définis en 2.1) → endpoint interne (first-party, pas de SDK tiers dans le bundle initial) ; le CMP (TCF) conditionne pub **et** analytics non essentiels ; per-event schema Zod (qualité des données Phase 9).

## Hypothèses signalées

- **H63** : Next.js 15 + Tailwind v4 + Radix + TanStack Query = stack proposée (§1) — c'est **la** décision structurante de cette validation.
- **H64** : hébergement front pressenti sur free tier compatible Next (Vercel Hobby ou Cloudflare Workers/OpenNext) — choix formalisé en 6.0/6.1 avec le reste de l'infra (cohérence R2/Workers D7).
- **H65** : pas de monorepo (une seule app Next contenant front + BFF + admin) tant que l'équipe est unique — simplicité d'abord (D1/H2).
