# Phase 6.0 — Best Practices Summary : Back-end

> **Étape** : 6.0 (recherche) · **Statut** : ✅ Validé HITL (D33)
> **S'appuie sur** : D5 (APIs), D6 (flags runtime sans redéploiement), D7 (streaming R2/HLS), D8/D11 (ingestion & UGC), D26 (front server-only), H70/H77/H78 (dettes du front à résorber)
> **Contrainte cadre** : exclusivement des services **gratuits ou à plan gratuit durable** (brief), extensible sans réécriture.

## 1. Architecture générale — BFF dans Next.js

**Proposition : pas de serveur séparé.** Le back-end v1 vit dans l'application Next.js déployée sur Vercel :

- **Route Handlers** (`app/api/*`) pour les endpoints appelés par le client (heartbeat lecteur, recherche instantanée) et les webhooks ;
- **Server Actions** pour les mutations de formulaires (contact, Ma liste, studio) — progressive enhancement natif, pas d'API publique à exposer ;
- Les pages continuent d'appeler les fonctions serveur directement (pattern `lib/*` server-only du front, D26) — le « BFF » est une **couche de fonctions**, pas un réseau de microservices.

**Pourquoi** : zéro coût, zéro latence inter-services, déploiement atomique front+back, et la migration ultérieure (worker dédié pour l'ingestion) est prévue par contrat d'interface. **Limite assumée** : pas de tâches > 60 s sur le plan gratuit Vercel (voir §5 ingestion).

## 2. Base de données — Postgres serverless (Neon)

| Option | Plan gratuit | Verdict |
|---|---|---|
| **Neon (Postgres)** | 0,5 Go, autosuspend, branches | ✅ **Proposé** : vrai Postgres, SQL standard, scale-to-zero |
| Supabase (Postgres + Auth + Storage) | 500 Mo, 2 projets, pause après 7 j d'inactivité | Alternative sérieuse (tout-en-un) ; la pause d'inactivité est un risque en démarrage |
| Turso (SQLite edge) | Généreux | Écosystème plus jeune, migrations Postgres plus tard = friction |
| Vercel Postgres | Facturation intégrée Vercel (Neon en marque blanche) | Équivalent Neon avec lock-in de facturation |

- **ORM : Drizzle** (léger, SQL-first, migrations versionnées dans le dépôt, excellent en serverless) ;
- Schéma v1 : `users`, `sessions`, `lists` (favoris/en-cours/historique — reprend le contrat du store local H70), `videos` (UGC + catalogue ingéré : statut `draft→processing→pending_review→published/rejected`, licence, déclaration de droits), `moderation_events`, `app_config` (flags D6), `contact_messages` (+ motif takedown) ;
- Pooling : driver HTTP de Neon (pas de pool TCP à gérer en serverless).

## 3. Authentification — Auth.js v5 (NextAuth)

- **Auth.js v5** auto-hébergé dans Next.js : gratuit, sans plafond d'utilisateurs, données chez nous (RGPD simple), adaptateur Drizzle ;
- v1 : **e-mail + mot de passe (argon2) et magic link** ; OAuth (Google) en option dès que les clés existent ; sessions JWT + rotation ;
- Écarté : Clerk/Auth0 (plafonds et coûts dès la traction, données tierces), Supabase Auth (retenu seulement si l'option Supabase globale l'emporte) ;
- **Migration H70** : à la première connexion, le client pousse le contenu de `cineplus.library.v1` → fusion serveur (dédoublonnée, le serveur fait foi ensuite) — le contrat du store front ne change pas (promesse Lot 3).

## 4. Flags runtime — lever la contrainte « redéploiement » (D6)

Exigence ferme D6 : pub ON/OFF **à effet immédiat sans redéploiement**. Les flags passent des variables d'env à la table `app_config` :

- lecture serveur avec cache mémoire court (~30-60 s) + invalidation par `revalidateTag` à l'écriture back-office ;
- l'API front ne change pas (`isAdsEnabled()` etc. deviennent async-DB — elles le sont déjà) ;
- env vars conservées comme **valeurs d'amorçage** et kill switch d'urgence (priorité : env `false` > base).

## 5. Ingestion vidéo (D7/D8) — la pièce exigeante du free tier

**Stockage/diffusion : Cloudflare R2** (10 Go gratuits, **zéro frais de sortie** — décisif pour la vidéo) + Worker de service des segments HLS avec **URLs signées HMAC** (D7).

**Encodage ffmpeg → HLS 3 qualités** : impossible dans Vercel/Workers (CPU/durée). Options gratuites comparées :

| Option | Verdict |
|---|---|
| **GitHub Actions comme worker d'encodage** (job déclenché par webhook, ffmpeg préinstallé sur les runners, upload direct R2) | ✅ **Proposé pour v1** : 2 000 min/mois gratuites, ~15-30 min de runner par film — dimensionné pour un catalogue naissant modéré a priori (D11) |
| Encodage navigateur (ffmpeg.wasm) | Écarté : 10-20× temps réel, onglet ouvert obligatoire, qualité incontrôlable |
| VPS free tier (Oracle/Fly) | Écarté v1 : maintenance d'une machine, quotas révocables ; plan B documenté |

Flux : upload direct navigateur → R2 (URL présignée, quota par utilisateur D11) → enregistrement `videos.processing` → dispatch encodage → dépôt HLS dans R2 → `pending_review` → modération (7.1) → `published` (visible dans /gratuit et /regarder). **Trajectoire** : le jour où les minutes GitHub saturent, le même contrat passe sur un worker dédié — aucune autre couche ne change.

## 6. Sécurité & anti-abus

Rate limiting sur mutations (fenêtre glissante en base — pas de Redis payant) · honeypot + délai minimal sur les formulaires (D13) · validation Zod à toutes les frontières (pattern D26 déjà en place) · URLs vidéo signées à expiration courte (D7) · en-têtes de sécurité (CSP compatible player/TMDB images, HSTS) · secrets uniquement en env Vercel · audit trail des actions de modération (7.1).

## 7. Observabilité & CI

Logs structurés JSON sur les route handlers (visibles dans Vercel) · `/api/health` (base + R2 + TMDB) · budget d'erreur simple avant vrai APM (Phase 9) · **CI GitHub Actions** : lint + build + Playwright (Lot 6) + Lighthouse CI (H79) sur chaque PR — la suite du Lot 6 devient bloquante.
